# 技术架构

## 1. 分层

### Web 壳层

负责页面容器、加载状态、错误提示和未来的设置菜单。它不直接承载玩法逻辑。

### Phaser 场景层

- `BootScene`：生成调试纹理、初始化共享状态。
- `GameScene`：负责战斗世界、实体生命周期和胜负流程。
- `UIScene`：只读取共享状态并绘制 HUD。

场景之间通过 Phaser Registry 和事件通信，避免 UI 直接操作战斗实体。

### 实体层

玩家、敌人和投射物负责自身最小行为。复杂规则不堆积在实体类中。

### 系统层

后续独立实现：

- 刷怪与波次
- 主动攻击和武器
- 经验与升级
- 待办增长
- 摸鱼区域
- Boss 阶段
- 掉落物与对象池
- 局外存档

### 数据层

难度、武器、敌人、进化配方和关卡时间表均使用 TypeScript 数据对象描述。内容调整不应要求修改场景主循环。

## 2. 状态边界

### 单局状态

- 心态
- 待办
- 怒气
- 经验与等级
- 击杀数
- 游戏内时间
- 当前难度
- 当前阶段：战斗、撤离、胜利或失败

单局状态在重新开始时完全重置。

### 永久状态

- 已解锁职业
- 已解锁武器
- 离职念头
- 成就与图鉴
- 设置项

永久状态后续通过版本化的本地存档适配器写入 `localStorage`。业务代码不直接调用浏览器存储。

## 3. 事件建议

- `session:started`
- `player:damaged`
- `player:defeated`
- `enemy:defeated`
- `xp:changed`
- `level:up`
- `backlog:changed`
- `rage:changed`
- `evacuation:started`
- `session:won`

事件负载必须有明确 TypeScript 类型，避免使用无结构字符串对象。

## 4. 性能策略

- 敌人和投射物使用对象池，减少频繁创建与销毁。
- 远距离敌人降低更新频率。
- 同屏敌人数设置软上限，刷怪器根据帧率降载。
- 攻击碰撞优先使用简单圆形或矩形。
- 大范围伤害使用空间分区或有限候选集。
- 特效和伤害数字设置数量上限。
- 默认目标为桌面端 60 FPS，低性能模式目标为稳定 30 FPS。

## 5. 数据驱动示例

```ts
interface WeaponDefinition {
  id: string;
  name: string;
  cooldownMs: number;
  damage: number;
  projectileSpeed: number;
  maxLevel: number;
}
```

武器行为与数值定义分离。新增武器时优先组合已有行为模块，而不是复制完整攻击代码。

## 6. 可测试边界

优先为以下纯逻辑编写单元测试：

- 难度倍率
- 经验阈值
- 工作时间换算
- 待办增长
- 升级候选抽取
- 武器进化条件

场景渲染和输入通过浏览器集成测试验证，不强行在单元测试中模拟整个 Phaser。

## 7. 后续演进

当 `GameScene` 超过可维护范围时，按系统顺序迁出逻辑。不要过早引入完整 ECS；只有当实体组合和性能数据证明需要时，再评估 ECS 或空间索引方案。
